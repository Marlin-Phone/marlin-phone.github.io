## 起因

我把服务器从阿里云迁移到了京东云，顺便把 AI 助手从 v1.x 升级到了 v2.0.0。结果发现助手回复消息变得极其缓慢——有时候一条消息要等 5 分钟才收到回复。

一开始我以为是大版本升级导致的 Bug，或者模型推理变慢了。但排查后发现，**问题跟代码无关，跟模型无关，纯粹是网络问题。**

## 排查过程

### 第一轮：直觉陷阱

我第一个反应是"模型推理变慢了"——每次助手回复前都要调大模型，多轮 tool call 串行执行，每次推理都要重新等模型返回，加起来自然就慢了。

但这个推理被一个关键信息推翻了：**之前从来没这么慢过，换服务器 + 升版本后才出现的。** 如果仅仅是模型推理慢，那应该在旧服务器上也一样慢才对。

### 第二轮：看日志

我让助手去查运行日志，发现了大量这样的错误：

```
ConnectionTimeoutError: Connection timeout to host 
https://api.sgroup.qq.com/v2/users/.../messages
```

这不是模型推理慢，而是**消息发不出去！** 助手早就推理完了，但往 QQ 的 API 发消息时连接超时了。

### 第三轮：拆解网络

为了搞清楚到底是哪一步出了问题，我设计了几个测试来隔离变量：

| 测试 | 目的 | 方法 |
|------|------|------|
| 测试1 | 检查 TCP 连接 | 纯 socket 连接，不涉及 TLS |
| 测试2 | 检查 TLS 握手 | socket + ssl 完整握手 |
| 测试3 | 走代理对比 | 通过代理服务器访问 |
| 测试4 | 排除软件干扰 | 用 curl 空环境变量直连 |

### 第四轮：实锤

结果非常清晰：

```
腾讯QQ API:  TCP=0.007s ✅  →  TLS=❌ 8.0s 超时
腾讯官网:    TCP=0.025s ✅  →  TLS=0.043s ✅
百度:        TCP=0.010s ✅  →  TLS=0.040s ✅
```

**TCP 三次握手 7 毫秒就通了，但在 TLS 握手阶段卡死了 8 秒。**

更离谱的是，这个问题是**间歇性发作**的——同一台服务器，同一套代码，同一个 API，有时候 50 次测试全部成功（平均 0.08s），有时候 20 次测试失败 18 次（成功率 10%）。

## 关键知识：TCP 三次握手 vs TLS 握手

很多人容易把这两个概念搞混，但其实它们是完全不同的两个阶段，**一个负责"能不能通"，一个负责"安不安全"**。

### TCP 三次握手（传输层）

当你的程序调用 `socket.connect()` 时，操作系统内核会帮你完成三次握手：

```
客户端                         服务器
  |        SYN (seq=x)          |
  | ──────────────────────────> |  ① 客户端：喂，我要连你
  |     SYN+ACK (seq=y, ack=x+1) |
  | <────────────────────────── |  ② 服务器：收到，我能连
  |     ACK (ack=y+1)           |
  | ──────────────────────────> |  ③ 客户端：好的，连上了
  |
  ▼ 连接建立成功 ✅
```

**关键特点：**
- 在操作系统内核中完成，纯内核态，不涉及用户态程序
- 不涉及证书、不涉及加密、不涉及任何业务逻辑
- 耗时极短，通常 0.00x~0.0x 秒
- 只要 IP 可达、端口开放，基本秒通

### TLS 握手（安全层）

TCP 连接建立好之后，**在 TCP 的通道上再走一套流程**，目的是加密通信。这完全在用户态完成，由 SSL 库（如 OpenSSL）实现：

```
客户端                         服务器
  |  ClientHello                 |
  |  (TLS版本, 密码套件列表)      |
  | ──────────────────────────> |  ① 客户端：我要加密，支持这些算法
  |  ServerHello + Certificate   |
  |  (选定算法 + 服务器证书)      |
  | <────────────────────────── |  ② 服务器：用AES-GCM，这是我的证书
  |  密钥交换                    |
  | ──────────────────────────> |  ③ 客户端：验证证书，生成密钥
  |  Finished                    |
  | <────────────────────────── |  ④ 服务器：密钥确认，开始加密
  |
  ▼ 安全通道建立 ✅
```

**关键特点：**
- 在用户态完成，涉及 SSL 库的复杂运算
- 需要证书验证、加密算法协商、密钥交换
- 涉及多个网络往返（RTT）
- 正常耗时 0.0x~0.x 秒
- **中间网络设备（防火墙、DPI）可能会干预这个过程**

### 两者的本质区别

| | TCP 三次握手 | TLS 握手 |
|------|------------|---------|
| 层级 | 传输层（OSI 第4层） | 表示层/会话层（OSI 第5~6层） |
| 实现位置 | 操作系统内核 | 用户态 SSL 库 |
| 目的 | 建立连接 | 加密通信 |
| 耗时 | 0.00x 秒 | 0.0x~0.x 秒 |
| 网络往返 | 1.5 个 RTT | 2~3 个 RTT |
| 可被中间设备干扰 | 通常不能 | **容易被防火墙/DPI 检测** |

**类比：**

TCP 就像**打电话**——你拨号，对方接听，说"喂？喂！"，确认双方都在线，通话通道就建好了。

TLS 就像**在电话上加了一个加密器**——你告诉对方"我用 AES-256 加密，这是我的密钥"，对方确认"收到，开始加密"。之后你们说的每一句话都是加密的，中间人听不到。

**但问题来了：如果中间有个接线员（防火墙/DPI），他可能不认识这个加密器，说"等一下，让我检查一下这是什么"，然后一直卡住不放行。**

## 根因分析

回到这次的问题。京东云到腾讯云之间可能经过了某些对 TLS 握手包做深度检测的中间设备，这些设备在检测 TLS ClientHello 时偶尔会卡住，导致握手包丢失或延迟。

```
TCP 三次握手: 0.007s ✅  →  连接建立成功
    ↓
TLS ClientHello: 我发出去了，但对方没收到 ❌
    ↓
等 10 秒超时 → 报错
```

**为什么百度、腾讯官网没问题？** 因为它们走的 CDN 节点不同，网络路径不同，经过的中间设备也不同。

**为什么阿里云没问题？** 阿里云到腾讯云的业务量大，有专门的优化路由和专线。京东云到腾讯云的流量小，可能被路由到了质量较差的链路上。

## 如何排查这类问题

如果你也遇到类似的问题——"TCP 能通，但 HTTPS 间歇性超时"，可以按这个思路排查：

### 第一步：隔离变量

写一个简单的测试脚本，分别测以下三个指标：

```python
# 测 TCP 连接（不涉及 TLS）
s = socket.socket()
s.settimeout(5)
s.connect((host, 443))  # 只测 TCP，不测 TLS

# 测 TLS 握手（在 TCP 之上做完整握手）
ctx = ssl.create_default_context()
tls_sock = ctx.wrap_socket(sock, server_hostname=host)  # 测 TLS
```

### 第二步：对比测试

**同一台服务器，对比不同的目标：**

| 目标 | 预期结果 |
|------|---------|
| 出问题的 API | 可能超时 |
| 大型网站（百度、腾讯官网） | 应该正常 |
| 同运营商的其他服务 | 视情况而定 |

如果只有特定目标出问题，说明是**目标端或中间链路**的问题，不是服务器自身网络问题。

### 第三步：走代理对比

如果能通过代理绕过直连路由并且问题消失，那就**实锤了直连路由的问题**。

### 第四步：持续监控

网络问题往往是**间歇性**的，不一定每次都能复现。写一个后台监控脚本，每 30 秒测一次，自动记录失败证据：

```python
while True:
    result = test_tls(host)
    if not result["success"]:
        save_evidence(result)  # 记录失败时间、耗时、错误类型
    time.sleep(30)
```

等收集到足够的失败数据后，再带着证据去联系云服务商。**没有数据，客服只会说"我们这边测是正常的"。**

## 总结

这次排查最大的教训是：**不要被表象迷惑，先看日志。**

- ❌ 直觉：版本升级有 Bug / 模型推理慢
- ✅ 真相：京东云到腾讯云的 TLS 握手间歇性抽风

通过隔离变量、对比测试、走代理验证三步，最终定位到了根因——**TCP 通但 TLS 不通，说明网络层没问题，问题出在安全层的中间设备上。**

这也是跨云服务商架构中一个容易被忽视的风险点：**不同云厂商之间的网络链路质量参差不齐，尤其是 TLS 握手这种需要深度包检测的场景，更容易受到中间设备的影响。**
