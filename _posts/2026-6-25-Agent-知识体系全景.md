## 一、写在前面

这篇文章不是教程，是一次**知识收网**。

过去几天我花了几个小时跟我的 AI Agent 对话，从 Agent 八股文一路聊到生产落地。这篇文章把对话中涉及的所有核心概念串联起来，形成一个完整的知识体系。

## 二、三层架构

Agent 系统的所有概念可以归纳为三个层次：

```
┌────────────────────┐
│   Workflow 模式     │  ← 多步骤编排
│  Chain / Router     │
│  Parallel / Orch    │
├────────────────────┤
│   Agent 架构        │  ← 想 + 做的循环
│  ReAct / Reflexion  │
│  Plan-and-Solve     │
├────────────────────┤
│   推理模式           │  ← 一步怎么想
│  Direct / CoT       │
│  ToT / Self-Con     │
└────────────────────┘
```

这三层是**堆叠关系**：上层依赖下层。

## 三、第一层：推理模式

### 什么是推理模式

推理模式 = LLM 在**一步思考单元**内部组织思维的方式。它不关心外部行动、工具调用、步骤编排——它只关心脑子里面怎么转。

### 真正的推理模式

| 模式 | 本质 | 一句话 |
|------|------|--------|
| Direct | 输入 → 输出 | 一步到位 |
| CoT（思维链） | 输入 → 推理1 → 推理2 → ... → 输出 | 一步步想 |
| Self-Consistency | 多条 CoT → 投票 | 多次思考取共识 |
| ToT（思维树） | 同时探索多条路径 → 剪枝 | 像下棋一样推演 |
| GoT（思维图） | 推理节点形成图结构 | 最灵活，最贵 |

### 核心洞察

CoT、ReAct、Reflexion 不是三个不同的推理模式。这是市面上八股文最常见的错误。

拆开来看：
- ReAct 里的 "Thought" 用的就是 CoT
- Reflexion 里的 "审查" 用的也是 CoT

它们共享同一个元模式：

```
当前状态 → LLM 评估 → 决定下一步 → 执行 → 新状态 → 再评估 → ...
```

区别只在于关注的方向不同：

| 模式 | 关注方向 | 类比 |
|------|---------|------|
| CoT | **向内看** — 盯着自己的推理链 | 心里盘算 |
| ReAct | **向外看** — 盯着外部世界的反馈 | 边做边看 |
| Reflexion | **向后看** — 盯着自己的历史输出 | 交卷后检查 |

它们是**同一个引擎在三个方向上的投影**，不是三个不同的引擎。

## 四、第二层：Agent 架构

### 架构不是平级的——它们是不同维度的

很多人学 Agent 时会看到一张并列的表格：

| ReAct | 边想边做 |
| Reflexion | 做完回头审查 |
| Plan-and-Solve | 先计划后执行 |
| Multi-Agent | 多 Agent 协作 |

但事实是：**它们不在同一个维度上，不能并列。**

正确的理解方式如下。

### 执行策略：ReAct vs Plan-and-Solve

这两个才是真正同一层级的东西——它们回答**"怎么执行"**：

```
ReAct：边想边做，没有预设计划
    Thought → Action → Observation → Thought → ...

Plan-and-Solve：先制定完整计划，再逐步执行
    Plan 阶段：制定计划，不写代码
    Solve 阶段：按计划执行，不改计划
```

**它们不是互斥的，可以嵌套。**

Copilot Workspace 就是例子：
- **外层**是 Plan-and-Solve：先规划 Issue 的修改方案，再执行
- **内层**的 Solve 阶段里嵌了 **ReAct**：写代码 → 检查语法 → 发现错误 → 修正

```python
# Plan-and-Solve 外壳
def copilot_workspace(issue):
    plan = make_plan(issue)          # Plan
    for step in plan:
        execute_step(step)           # Solve，内部是 ReAct

# ReAct 内核
def execute_step(step):
    while not done:
        thought = llm.think(...)
        if thought["action"] == "write":
            write_file(...)
        elif thought["action"] == "check":
            if has_error: continue   # 自己修
```

### 质量机制：Reflexion（可叠加）

Reflexion 不是跟 ReAct、Plan-and-Solve 平级的——它是**可以贴在任意执行策略外面的质量保障层**：

```
                   无 Reflexion              有 Reflexion
                   ────────────              ────────────
ReAct              ChatGPT 普通对话          ChatGPT 写完代码自己检查一遍
Plan-and-Solve    按计划机械执行              执行完审查 → 发现不对 → 修订计划 → 重做
```

Reflexion 回答的是"**怎么保证质量**"，而不是"怎么执行"。

### 协作模式：Multi-Agent（另一个维度）

Multi-Agent 回答的是"**一个人干还是多个人干**"，跟"怎么干"是正交的：

```
             单 Agent                 Multi-Agent
             ────────                ────────────
ReAct        ChatGPT                 多个 ChatGPT 各干各的
Plan-and-    一个 Agent 计划并执行     一个 Orchestrator 分配任务
Solve                                 多个 Specialist 各自执行
```

### 总结：真实的架构关系

```
执行策略（同一级，可二选一或嵌套）
    ├── ReAct（边想边做）
    └── Plan-and-Solve（先想后做）

质量机制（可叠加在任一种之上）
    └── Reflexion（执行后审查）

协作模式（另一个维度）
    └── Multi-Agent（多个 Agent 协作）
```

**不能把它们列成一个四选一的表格——那不是它们真实的关系。**

## 五、第三层：Workflow 模式

Workflow = 把 Agent 架构 + 工具调用 + 决策逻辑编排成可执行的流程。

### 常见 Workflow 模式

| 模式 | 流程是否固定 | 一句话 |
|------|------------|--------|
| **Chain（链式）** | ✅ 完全固定 | 一步一步走 |
| **Router（路由）** | ⚠️ 分支固定，选谁不一定 | 先判断再分流 |
| **Parallel（并行）** | ✅ 多个同时 | 多路齐发，汇总结果 |
| **Orchestrator（编排器）** | ❌ 完全动态 | 主控 LLM 动态分配 |

### 生产环境 vs 八股文

八股文里的 Workflow：LLM 动态决策每一步做什么。

生产环境里的 Workflow：**队列 + 线程池 + 显式状态管理 + 补偿机制**。

每一步之间用队列解耦，每个阶段有独立线程池，数据存在显式的状态管理里而非 LLM 上下文里。保证毫秒级延迟和 100% 可预测性。

## 六、落地案例：Agent 自动提交 GitHub

### 架构

```
我说"写一篇博客"
    → Agent 生成 Markdown（Jekyll 格式）
    → 通过 GitHub Contents API 写入 _posts/
    → GitHub Actions 自动构建部署
```

### 关键设计决策

1. **API 直写，不用 git clone**：Agent 环境可能没有 Git 配置
2. **Human-in-the-Loop**：生成内容先给人审查，批准后才推送
3. **Skill 固化**：把流程写进 SKILL.md，Agent 重启后能力不丢失
4. **Conventional Commits**：commit message 遵循 `feat: 添加xxx博客` 规范

### 踩坑记录

| 坑 | 表现 | 修复 |
|----|------|------|
| 时区 | 用了 UTC，差 8 小时 | 强制先获取上海时间 |
| Git 规范 | commit 写了 `博客: xxx` | 改用 `feat: 添加xxx` |
| 审批缺失 | 直接推送，没给人看 | 加入 Human-in-the-Loop 节点 |

## 七、整个体系的关系图

```
┌─────────────────────────────────────────────┐
│                 Workflow                     │
│      Chain / Router / Parallel / Orchestrator│
├─────────────────────────────────────────────┤
│             Agent 架构                       │
│   执行策略: ReAct / Plan-and-Solve           │
│   质量机制: Reflexion (可叠加)                │
│   协作模式: Multi-Agent (另一维度)            │
│         ┌───────────────────┐                │
│         │  推理模式           │                │
│         │ CoT / ToT / Direct │                │
│         └───────────────────┘                │
├─────────────────────────────────────────────┤
│           能力底座                            │
│     Tools / MCP / Function Calling           │
└─────────────────────────────────────────────┘
```

从下往上：
- **Tools / MCP** 决定 Agent 能做什么
- **推理模式** 决定每一步怎么想
- **Agent 架构** 决定想和做的循环 + 质量保障 + 协作方式
- **Workflow** 决定整个流程的编排

## 八、总结

整个知识体系可以浓缩为三句话：

> 1. **推理模式**管一步怎么想（CoT / Direct）
> 2. **Agent 架构**管怎么执行 + 怎么保证质量 + 谁来做
> 3. **Workflow** 管多步的编排（Chain / Orchestrator）

这三层是堆叠关系，不是替代关系。生产系统里它们**同时存在**：
- 高频确定场景用 Chain Workflow 编排步骤
- 每一步内部用确定性代码执行（而不是 CoT）
- 复杂决策时用 ReAct Agent 做深度分析

**八股文给你概念，生产环境给你约束。把概念放进三层体系里理解，你就知道什么时候该用什么了。**
